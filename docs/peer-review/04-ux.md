# 04 — User experience

## 4.1 Mental model

The user is a **photographer, not a developer.** They want to:

1. Drop a folder of RAFs/NEFs into the app.
2. See a contact-sheet of thumbnails with quality info baked into the chrome.
3. Hit `K` / `M` / `R` on each photo, with the keystroke moving the file *and* writing a sidecar rating that downstream apps will read.
4. Optionally collapse near-duplicate bursts down to one keeper.
5. Optionally run a "score-blind" training round so the app learns their taste.

**The contract:** "your judgment is the boss; the app's job is to surface signal and remove friction."

Concretely, the app must never:
- Make a decision for the user without an explicit auto-cull confirmation.
- Hide *why* a photo scored low (DetailView is always one keypress away).
- Be color-loud — chrome must stay neutral so the photo's own colour is judged honestly (see §4.6 design system).

## 4.2 The five views

```
┌──────────────────────────────────────────────────────────────┐
│  TabBar                                       Header chrome  │
├──────────────────────────────────────────────────────────────┤
│  [Grid] [Groups] [Train · N]      ← view switcher            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌────────────────────────────────────────────────────┐     │
│   │                                                    │     │
│   │  Main view (one of: Grid · Groups · Train ·        │     │
│   │  DetailView overlay · empty-tab CTA)               │     │
│   │                                                    │     │
│   └────────────────────────────────────────────────────┘     │
│                                                              │
│   [Sort ▾] [Size ▾]                  [Folders] [Settings ⚙]  │
│   floating bottom toolbar (hidden when DetailView is open)   │
└──────────────────────────────────────────────────────────────┘
```

| View | When | What |
|---|---|---|
| **Empty** | New tab, no folder picked | Centred "+ New analysis" button — opens Finder picker |
| **Grid** | Default after analysis | Contact-sheet of thumbnails, columns 2–8 (user-tunable) |
| **DetailView** | Space / click on grid cell | Full-screen image + scrollable info panel with all scores |
| **Groups** | Tab switch | Filmstrip rows, one per SigLIP cluster, hero gold-ringed |
| **Train · N** | After ≥1 undecided photo | Score-blind queue: one photo at a time, no scores visible, K/M/R |

Plus six modals: **Settings**, **Shortcuts**, **AutoCull preview**, **Conflict** (training-mode disagreement), **Confirm** (destructive bulk actions), and the inline **Tab close confirmation**.

## 4.3 Keyboard model — the spine of the app

The app is keyboard-first. **Every action a user does in a culling session is reachable from the keyboard.** This is because culling 500 RAFs over an hour is unbearable with mouse-only interaction.

Bindings (from `App.jsx` `useHotkeys`):

| Key | Action | Where |
|---|---|---|
| `K` | Decision = Keep | Grid + DetailView + Train |
| `M` | Decision = Maybe | Grid + DetailView + Train |
| `R` | Decision = Reject | Grid + DetailView + Train (`X` was original; renamed `R` in shortcuts modal) |
| `←` `→` | Previous / next photo | Grid + DetailView + Group focus |
| `↑` `↓` | Up / down by `cols()` rows | Grid only |
| `Space` | Toggle DetailView | Grid |
| `Esc` | Clear compare → clear focus → close DetailView | DetailView, Groups |
| `H` | Promote focused photo to group hero | Groups (focused photo only) |
| `Tab` / `Shift+Tab` | Next / previous similarity group | Groups |
| `⌘Z` | Undo last decision | Anywhere; toast confirms |
| `=` / `-` | Decrease / increase grid columns | Grid |

**Hotkey discipline:**
- K/M/R/arrows are **disabled in Groups and Train mode contexts where they would be ambiguous** (e.g., arrow keys move group focus, not grid focus, when a group is focused).
- All bindings respect `react-hotkeys-hook`'s default that *no* hotkey fires while an `<input>` or `<textarea>` is focused.

**Discoverability:** the Shortcuts modal (?-key in keyboard guides) lives behind a header button, not a hotkey, on purpose — to be reachable from a mouse-only first session.

## 4.4 The DetailView panel — the "why" surface

DetailView is where the ML signal becomes legible. The panel is **sticky-headed, scrollable**, with three kinds of section components:

1. **`Section`** (collapsible, default closed) — score sections with a headline chip showing the headline number. On expand, draws a bordered chrome around the body.
2. **`AiQualityRow`** (non-collapsible) — for IQA / aesthetic / face quality. Uses **colour-tinted chips** mapping the TOPIQ band to a tint:
   - sage (`#7DB89A`) for "Excellent" (≥75)
   - cyan (`#5BB8D4`) for "Good" (≥55)
   - amber (`#E8B84A`) for "Fair" (≥35)
   - coral (`#C97B7B`) for "Poor" (<35)
3. **`CollapsibleSection`** (default open) — for EXIF, narrative explanation. Chevron right-aligned.

Section open/closed state is **persisted per-section** in `localStorage` (`pca.detail.section.<id>`).

**LearnedSignals** (the stacked bar visualising what drove the personal-model delta):
- Up to top-3 features by absolute contribution, plus an aggregate "other N signals" segment in **Muted Bronze (`#8C7A5E`)**.
- Direction encoded in segment position (left of zero = pulling down, right = pulling up).
- Only renders when the personal model is ready *and* this image has a `personal_score` distinct from `overall_score`.

**Hover popovers (InfoTooltip + HoverPopover):**
- Both use `createPortal(content, document.body)`. Required because ancestor `hover:opacity-*` cascades will dim the popover otherwise — CSS opacity flows to children, only portaling escapes it. (See `feedback_hover_popover_portal` in author memory.)

**Histogram + clipping:**
- Lazy-loaded on first expand of the histogram section.
- Two toggle buttons reveal *transparent PNG overlays* fetched from `GET /clipping-mask/{id}?mode=highlights|shadows`. Amber for highlights, cyan for shadows.
- Overlay uses `mix-blend-mode: screen` so the underlying photo stays visible.

## 4.5 The tab system

The tab system is the load-bearing architectural-UX choice. It exists to support the author's actual workflow of culling **two folders concurrently** (e.g., a wedding shoot in /Day1 and /Day2, switching context to compare).

**Rules:**
1. **Each tab is a workspace bound to one source folder.** Independent images list, progress bar, selection, sort/filter state, watch-live flag.
2. **Sequential analysis only.** Starting analysis on tab Y while tab X is analysing prompts a "another analysis is in progress" modal; user must Stop X first.
3. **The rightmost tab is always an empty "New analysis" stub.** Non-closable, non-draggable. Clicking it opens the macOS folder picker directly (no intermediate empty-tab toolbar).
4. **Drag-reorder via HTML5 drag events.** The trailing tab blocks drops. Analysing tabs are not draggable.
5. **Watch live is per-tab (a `watchLive: boolean`).** Toggling on tab X turns it off everywhere else (single-watcher backend). Tab strip shows a flashing cyan dot on the watching tab; hover tooltip on every other tab tells the user where it's active.

**Persistence:**
- Tabs are reconstructed on launch from `GET /folders` (one tab per folder with analysed images) + `GET /watch` (detects active watcher folder).
- The active tab's `folderPath` is saved to `localStorage` (`pca.activeFolderPath`) and used to restore selection on next launch. UUIDs regenerate every session.

**Per-folder destinations (`TabFoldersPopover`):**
- Each tab has an opt-in K/M/X destination override stored in `folder_settings` table keyed by `source_folder`.
- Defaults to `{folder}/_Keeps`, `{folder}/_Maybes`, `{folder}/_Trash`.
- Pill in the bottom toolbar: "Folders" / "Folders: defaults" / "Folders: 2 custom" — surfaces whether overrides are active.
- A 2026-05-04 migration lifted the legacy global K/M/X settings into per-folder rows on first launch after upgrade.

## 4.6 Design system — what's enforced

**Source of truth:** `docs/DESIGN.md` (~36 KB). One-line summary: **Raycast-adapted dark theme with the brand red stripped out.** Photographers need their displays to be neutral so colour decisions on RAW files aren't biased by UI tint.

| Token | Value | Role |
|---|---|---|
| Canvas | `#07080a` | Page background — near-black, slightly blue-cold, sits below typical RAW shadow values |
| Surface 100 | `#101111` | Toolbar, panels |
| Surface 200 | `#161718` | Card surface (idle), DetailView header |
| Cool Cyan | `#5BB8D4` | Selection ring, focus ring, active tab underline |
| Warm Amber | `#E8B84A` | Maybe state + "best in group" badge |
| Keep Sage | `#7DB89A` | Keep state |
| Reject Coral | `#C97B7B` | Reject state |
| Personal Indigo | `#7B82C9` | Personal-score bar — sole exception to neutral score-bar fill |
| Muted Bronze | `#8C7A5E` | Aggregate / "other" segment in stacked charts |

**Score-bar colour rule:** *Neutral mid-grey (`#a8a8a8`) over a `#1b1c1e` track for everything except personal score.* Score bars never use red/green/yellow. The chrome stays color-neutral so the photo is always the loudest thing on the row.

**Hover discipline:** Hover effects are **opacity-only**, never background-colour. Maintains the "image is the only thing radiating colour" feel.

**Typography:** Inter @ +0.2px letter-spacing, weight 500 baseline. GeistMono for EXIF, scores, file paths.

**Hotkey badges:** styled as physical key caps with a 5-layer shadow (`--shadow-key`). Reinforces that the keys are real, hardware-pluggable controls.

**The `.label` class** (defined in `index.css`):
```css
.label {
  font-size: 14px;
  font-weight: 500;
  letter-spacing: 0.002em;
  color: var(--color-text-secondary); /* #cecece */
  text-transform: none; /* title case, not all-caps */
}
```
Used app-wide for section/panel headers. Earlier prototypes had 11px ALL-CAPS bronze labels; that was rejected as "interface noise" — see `docs/DESIGN.md` §2.

## 4.7 Empty states & onboarding

**Empty new tab:** the `+ New analysis` tab opens the Finder picker directly on click. There is **no marketing-style hero state**. The author tried a centred "Welcome — pick a folder to begin" CTA and rejected it ("Hero empty states felt wrong here" — see `feedback_hero_empty_state` in author memory).

**First-run intro modal in Train mode:** gated by `localStorage.pca.trainingIntroSeen`. Shown the first time the user enters Train mode. Explains the score-blind concept and the conflict-resolution flow.

**No sign-up, no welcome tour, no analytics.** This is a personal tool.

## 4.8 Toasts & feedback

Two parallel toast layers:

**`DownloadToast`** (`frontend/src/ui/toasts.jsx`):
- Fixed bottom-right, single-purpose: model loading.
- Polls `/model-status` every 1 s. Shows model name + size + elapsed time.
- One row per model currently downloading.
- z-index 60.

**`ToastStack`** (general feedback):
- Fixed bottom-centre. z-index 65 (above DownloadToast).
- Coloured left-border accent: error (coral), success (sage), info (cyan), warning (amber).
- Optional action button (`label + onClick`) — used for the **Undo decision** toast.
- Auto-dismiss after 4 s unless `persistent: true`.

**Live-tab error stream:** when a tab has `watchLive: true`, polls `/debug/last-errors` every 5 s and toasts new errors (primed on entry to avoid replaying pre-existing errors).

## 4.9 Stop / cancel honesty

When the user hits **Stop analysis** during a batch:

1. Backend sets `_progress["stop_requested"] = True` and `_stop_event.set()`.
2. The current photo finishes its in-flight scorer (RAW decode is uncancellable). 2–5 s tail.
3. `quality_analyzer.py` raises `StopRequested` between scorer boundaries → `pool.shutdown(wait=False, cancel_futures=True)`.
4. UI button label flips to **"Stopping…"** in amber for the duration.

The "Stopping…" affordance is deliberately honest: it tells the user the system *heard* them but is finishing the current photo. Pretending the cancel is instant would be a lie.

## 4.10 Friction points the author already knows about

| Friction | Severity | Source |
|---|---|---|
| 2–5 s "Stopping…" tail surprises new users | Low | RAW decode is uncancellable; trade-off accepted |
| First analysis blocks for ~25 s while Phase 2 weights download | Medium | Mitigated by DownloadToast; reviewer should question whether a "sample first photo only" mode would soften this |
| Auto-cull is opaque about *which* rule fired | Medium | The preview shows counts (12 keep / 8 maybe / 30 reject) but not "of those 30 rejects, 5 were instant-rejected for closed eyes." |
| K/M/R hotkeys overload across views | Low | Disabled in contexts where they'd be ambiguous, but a new user can be surprised |
| "Reject all but best" 2-tap confirmation has 3 s timeout | Low | If the user gets distracted between taps, the action silently rearms |
| No explanation of *what* the personal model learned beyond top-3 features | Medium | LearnedSignals stacked bar shows directions but not, e.g., "your kept photos average 8 EV brighter than your rejects." |
| Settings → Model has 9 thresholds + 4 toggles in one tab | Medium | Sticky-footer pattern helps; cross-field validation (maybe < keep) prevents save errors. But the surface is dense |

A reviewer should look hardest at the auto-cull modal — it's the highest-stakes interaction (one click moves potentially hundreds of files) and its preview UI underplays the "30 rejects, all from one rule" failure mode.
